--- layout: post title: "The ChatScrypt Architecture Part III: Authenticated Inductive Diffie-Hellman" comments: true date: 2020-01-20 --- I came up with the Authenticated Inductive Diffie-Hellman (AIDH) protocol as I finished up Part II in this series, solving the issues I realized as I was analyzing the R2DH (it was very tempting to call this the R2D2 protocol). In addition, as far as I can tell, it has all the properties advertised of the Signal protocol on Wikipedia: "confidentiality, integrity, authentication, participant consistency, destination validation, forward secrecy, post-compromise security (aka future secrecy), causality preservation, message unlinkability, message repudiation, participation repudiation, and asynchronicity." Let's briefly go through them:
  • Confidentiality: Messages are encrypted and can only be read by Alice and Bob.
  • Integrity: If an attacker modifies a message ciphertext, it is completely destroyed.
  • Authentication: Alice can prove she has the private key of a long-term public key known to be Alice's, authenticating herself. (The public key needs to be verified by Bob externally, in real life.)
  • Participant consistency: For pairwise conversations, this is automatic from authentication.
  • Destination validation: To be honest I'm not sure what this means, but Alice eventually knows whether Bob received her message.
  • Forward secrecy: Old ciphertexts have keys that are not obtainable from future keys.
  • Future secrecy: The current key is not derivable from past keys.
  • Causality preservation: Information about the order of messages is encrypted along with the messages.
  • Message unlinkability: Breaking one message compromises none of the security properties of another message.
  • Message repudiation (deniability): Bob may publish a message but Alice may claim it was forged or modified.
  • Participation repudation (strong deniability): Same as message repudiation, but for whole conversations.
  • Asynchronicity: Alice can sent messages to a mailbox for Bob; he doesn't need to be there.

The AIDH protocol is simpler than the Signal protocol and achieves equivalent or superior security (aside from "security through obscurity," since a more complicated protocol is tautologically more obscure). I'm not entirely sure how much simpler though, because I'm not a fan of the published exposition for the Signal protocol. I will present AIDH using an inductive "finite state machine" type approach.

Alice's state

Once the protocol is initiated, inbetween messages Alice will have:

  • A list of integers enumerating which of Bob's messages she has received so far. (The contiguous part stemming from 0 may be deleted.)
  • A list of her most recent private encryption keys, with newest key x and oldest key x'.
  • A list of Bob's most recent public encryption keys, with newest key g^y mod N.
In non-pathological settings, these will have length 0, 2, and 1 respectively.

Initialization

Bob publishes to the server a new signed public key g^y mod N. He refreshes this key every time someone initiates a new correspondence with him. Alice publishes to the server an unsigned but long-term public key g^a mod N.

When Alice wants to initiate a correspondence with Bob, she fetches this public key, verifies the signature, and sends some metadata to Bob using the shared Diffie-Hellman encryption key

g^(ay) mod N.
Alice then initalizes her state:
  • She creates an empty array of integers.
  • Alice's private key list is initialized with x = a.
  • Bob's public key list is initialized with g^y mod N.
If roles are reversed and Alice receives the first message, she intializes as follows:
  • She creates a list of integers initialized with 1.
  • Alice's private key list is initialized with her published server key x.
  • Bob's public key list is initialized with g^b mod N.

Sending a message

Alice creates a new Diffie-Hellman key pair with private key x and encrypts her message using the shared Diffie-Hellman encryption keys
g^(xy) mod N and g^(x'y) mod N.
She attaches the following to her message:
  • (unencrypted) The public keys g^x, g^x', g^y mod N. The latter two are optional.
  • (encrypted) The message number, and for padding, a timestamp.

Alice then updates her state:

  • She pushes x to her list of recent private keys.

Receiving a message

The attached public keys in this case are g^y, g^y', g^x mod N. For authentication, she searches for y' in her list of Bob's public keys. She searches for x in her list of private keys. She decrypts the message and updates her state as follows if the message number and timestamp are sensible:
  • She pushes the message number to her list of integers. If the list is contiguous from the beginning, she can empty the list.
  • If the messages were contiguous, Alice may delete all of her private keys (strictly) before x. (Bob has used it, and from now on he will only use it or a newer key.)
  • Alice pushes the new key g^y mod N. If the messages were contiguous, she may delete all of Bob's public keys (strictly) before y'.
